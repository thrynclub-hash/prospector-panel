# Phase 02.1 Research: Reusar logo, paleta de cores e fotos do site original

**Researched:** 2026-07-08
**Requirement:** REDESENHAR-05
**Question answered:** What do I need to know to PLAN this phase well?

---

## TL;DR / Recommendation

1. **Logo:** extend the existing `captureScreenshot` Microlink call with `&meta=true`. Microlink's normalized metadata already returns a resolved `logo` object (`{ url, type, width, height }`) using the standard favicon/apple-touch-icon/og:image resolution chain — confirmed live against `stripe.com` (returned the 180×180 favicon PNG). **Zero new library, zero new HTTP round-trip** — same call already made for the "before" screenshot.
2. **Photos (gallery):** Microlink's `meta=true` only returns a *single* `image` field (an og:image-class social preview, not a gallery) — **not enough** for "hero/team photos." This needs a **separate lightweight scrape**: plain `fetch()` of the lead's site HTML + regex-based `<img>` extraction, following the exact pattern already established in `lib/email-scrape.ts` (timeout, User-Agent header, try/catch → `null`, no library). No HTML parser (cheerio etc.) is installed in this project; adding one is optional, not required — regex is consistent with existing convention and sufficient for a best-effort heuristic.
3. **Color palette:** Microlink returns **no color/theme field at all** (confirmed against docs + live response). Two-tier approach, no new heavy dependency needed:
   - Primary: regex-extract `<meta name="theme-color" content="#hex">` from the raw HTML fetched in step 2 (same fetch, free, many sites already declare this for mobile browser chrome).
   - Fallback: compute a dominant color from the extracted logo image using `sharp` — **already present in `node_modules`** (transitive dep of Next.js for `next/image`) but **not currently declared in `package.json`**; recommend adding it as an explicit direct dependency so it doesn't silently disappear on a future Next.js upgrade. All Route Handlers here run on the default Node.js runtime (no `export const runtime = "edge"` found anywhere in `app/`), so `sharp`'s native binary works fine.
   - If neither yields a usable color → omit the field entirely (`null`), which is the specified non-blocking fallback.
4. **Schema:** `redesigns.content` (jsonb) has **no CHECK constraint or JSON schema at the DB level** (confirmed in `supabase/migrations/20260708130000_redesigns.sql` — plain `content jsonb not null`). The "CONGELADO" freeze is a TypeScript/convention discipline only. **Adding a new optional field is safe at the DB layer.** The real risk is in application code that reconstructs the object explicitly instead of spreading it — see the critical finding below, which changes exactly *where* the new field should live in the type.
5. **AI Gateway:** **not needed.** This is a pure scrape/extraction feature — no new `generateObject` call, no prompt. It runs alongside (not instead of) the existing `generateRedesignCopy()` call.

## Critical finding: where the new field goes in `RedesignContent` (and why it matters more than the extraction mechanism)

`types/redesign-content.ts` is genuinely frozen only in the sense that 5 files touch `content.photos`/`content.facts`/`content.generated` directly, and **two of them reconstruct objects explicitly by field name rather than spreading unknown keys through**:

- **`app/painel/leads/[leadId]/redesenhar/editar/editor-form.tsx`** (line 79) — on every Editor save, the client sends:
  ```ts
  body: JSON.stringify({ generated, photos: { logoUrl: initialContent.photos.logoUrl, photoUrls } })
  ```
  This is a hand-built object literal with exactly two named keys. **Any third key added inside `photos` (e.g. a hypothetical `photos.themeColor`) would be silently dropped the first time a user opens the Editor and clicks "Salvar edição"** — not on generation, only on first edit.

- **`app/api/redesigns/[id]/route.ts`** (PATCH handler, line ~33-37) — on the server side, the update is:
  ```ts
  const updatedContent: RedesignContent = {
    facts: currentContent.facts,                    // always inherited from DB, never from body
    generated: body.generated ?? currentContent.generated,
    photos: body.photos ?? currentContent.photos,   // wholesale replace, not merged
  };
  ```
  Because `body.photos` is *always present* in the Editor's request (see above), `photos: body.photos ?? currentContent.photos` always takes the client's (field-stripped) version — it never falls through to `currentContent.photos`.

**Consequence for planning:** do **not** nest the new color field inside `photos`. Instead:

- Add it as a **new top-level sibling key** on `RedesignContent`, e.g. `theme: { primaryColor: string | null } | null` (optional/nullable so old DB rows without the key still satisfy the type when cast).
- In the PATCH route, thread it through **exactly like `facts`** — always inherited from `currentContent.theme`, never accepted from the PATCH body (there is no Editor UI for color, so it should never be client-settable): `theme: currentContent.theme`.
- This mirrors the *existing* pattern in the same file for `facts` (a field written once at generation time, then permanently read-only through the Editor) rather than the `photos` pattern (client-owned, round-tripped every save). Copy the right precedent.
- No changes needed to `editor-form.tsx` at all under this design, since color is never part of its save payload.

If a future implementer instead prefers nesting the new data inside `photos` (e.g. `photos.logoSourcedFrom` or similar), that is *also* fine **only if** the PATCH route's `photos` handling is changed from wholesale-replace to a shallow merge (`photos: { ...currentContent.photos, ...(body.photos ?? {}) }`). Either path works; **the thing that must not happen** is adding a new nested key under `photos` while leaving the PATCH route's `body.photos ?? currentContent.photos` logic untouched — that combination is the one that silently loses data.

**Full inventory of files touching `RedesignContent` fields directly** (grepped project-wide, 14 files reference the type; only these touch `.photos`/`.facts`/`.generated` specifically):
| File | Role | Risk |
|---|---|---|
| `types/redesign-content.ts` | type definition | — |
| `app/api/redesigns/generate/route.ts` | writer (creates `content` on generation) | none — just add the new key here |
| `app/api/redesigns/[id]/route.ts` | PATCH (Editor save) | **the wholesale-replace risk described above** |
| `app/painel/leads/[leadId]/redesenhar/editar/editor-form.tsx` | client save payload | **the field-stripping risk described above** |
| `app/painel/leads/[leadId]/redesenhar/redesign-preview.tsx` | renderer (read-only) | none — shared by both private preview and public demo, see below |
| `app/demo/[slug]/page.tsx` | public demo page | none — casts `content` and renders via the *same* `RedesignPreview` component, no separate rendering logic to keep in sync |
| `app/api/redesigns/[id]/photo/route.ts` | photo upload endpoint | none — only returns a URL, doesn't touch `content` |
| `app/painel/leads/[leadId]/redesenhar/comparator.tsx`, `compare/page.tsx`, `preview/page.tsx` | pass `content` through | none — confirmed no `.photos`/`.facts`/`.generated` field access |

`app/painel/leads/[leadId]/redesenhar/redesign-preview.tsx` is the **single shared rendering component** for both the private preview/comparator and the public `/demo/[slug]` page — one place to update to actually *use* the new logo/photos/color data (currently it already conditionally renders `content.photos.logoUrl` if present, falling back to a typographic monogram; it needs the same optional-chaining treatment for a new `theme.primaryColor`, e.g. using it as an accent color on headings/buttons instead of the hardcoded `slate-900`/`slate-600` neutral palette, falling back to the current neutral template when absent).

## Extraction mechanisms, ranked by reliability

### Logo
1. **Microlink `meta=true` (recommended, primary).** Confirmed live: `https://api.microlink.io/?url=...&meta=true` returns `data.logo = { url, type, width, height }` resolved via the standard metascraper-style priority chain (apple-touch-icon → shortcut icon → msapplication-TileImage → favicon.ico fallback, roughly — exact internal rule list isn't published in Microlink's docs, but the *output* was verified: Stripe resolved to their 180×180 favicon PNG). This can be added to the **same call already made in `lib/screenshot.ts`** — just append `&meta=true` to the existing URL and read `data.data.logo?.url` alongside `data.data.screenshot?.url`. No new API integration, no new cost tier (Microlink's free/anonymous tier is a flat "25 reqs/min" soft limit per their rate-limit docs — multiple output params on one URL is one HTTP call, not billed/counted per-feature).
2. **Fallback if Microlink's `logo` is null/API fails:** none needed for v1 — CONTEXT.md explicitly specifies falling back to today's behavior (typographic monogram) when extraction fails, and `redesign-preview.tsx` already implements exactly that fallback for `logoUrl === null`. Don't build a second logo-scraping mechanism; let it degrade.

### Photos (hero/team gallery)
1. **Direct HTML fetch + regex, following `lib/email-scrape.ts`'s exact pattern** (plain `fetch(url, { signal: AbortSignal.timeout(...), headers: { "User-Agent": ... } })`, try/catch → `null`/`[]`, never throws). Extract `<img ...src="...">` (and ideally `srcset`) occurrences, resolve relative URLs against the site's origin, and apply a cheap heuristic filter:
   - Reject by file-name pattern: `icon`, `sprite`, `favicon`, `logo` (already covered separately), tracking pixels (`1x1`, `pixel.gif`), common ad/CDN tracker domains.
   - Reject by declared `width`/`height` attributes below a threshold (e.g. < 150px) when present — many CMS themes annotate real content images with larger explicit dimensions than icons/decorative assets.
   - Prefer images inside likely-content containers if a cheap string-proximity heuristic is worth adding (e.g. appears near `<header>`, `class="hero"`, `class="team"` in the raw markup) — optional refinement, not required for v1; a simpler "take the first N images above the size threshold" is a reasonable v1 heuristic given the explicit non-blocking/best-effort framing in CONTEXT.md.
   - Cap to ~3 (matches the existing Places-photos cap of 3 in `generate/route.ts`).
2. **No HTML parser library is currently installed** (`cheerio`, `linkedom`, `node-html-parser` — none present). Regex is consistent with this codebase's existing convention (`email-scrape.ts` uses a raw regex against `html.match()`, not a DOM parser) and is proportionate to a best-effort, degrade-gracefully feature. Adding `cheerio` (~decent size, pure JS, no native deps, Vercel-friendly) is a reasonable upgrade if regex proves too fragile in testing, but is **not required** to satisfy the phase's success criteria.
3. Each selected photo URL must be **re-hosted**, exactly like Places photos are today (`uploadAsset`/`uploadFromUrl` in `lib/storage/assets.ts`) — never linked hotlinked from the lead's own site directly into `photoUrls`. **`uploadFromUrl(supabase, { userId, path, sourceUrl })` already exists in this file and is currently unused anywhere in the codebase** — it was seemingly written in anticipation of exactly this need (fetches an arbitrary URL, re-hosts to Supabase Storage, same try/catch → `null` degrade pattern). This is the correct function to call for both the scraped logo and the scraped photos; no new storage code needed.

### Color palette
1. **`<meta name="theme-color" content="#RRGGBB">` regex extraction from the same HTML fetched for photos** — free, deterministic, zero image processing, and a genuinely common convention (PWA/mobile browser chrome tinting) that a meaningful fraction of small-business sites/CMS themes (WordPress, Wix, Squarespace) emit by default.
2. **Fallback: dominant-color-from-logo-image via `sharp`.** `sharp` is already on disk (`node_modules/sharp@0.34.5`, transitive via Next.js's `next/image` optional dependency) but **not listed in `package.json`** — recommend adding it explicitly (`npm install sharp`) so file-tracing/bundling for the Vercel Function doesn't depend on an undeclared transitive package that could be dropped or changed version on a future `next` upgrade. Usage: `sharp(logoBytes).resize(32, 32).raw().toBuffer()` → iterate the small pixel buffer, bucket into coarse RGB buckets, exclude near-white/near-transparent/near-black pixels (common on logos with transparent or white backgrounds, which would otherwise average to a useless white/gray), pick the most frequent bucket. This avoids installing a dedicated color-quantization package (`node-vibrant`, `colorthief`) that typically pulls in `canvas`/`jimp` — heavier native/pure-JS image-decoding stacks than reusing the `sharp` binary already on disk. No dedicated palette npm package is necessary; a ~20-30 line manual quantization function against a `sharp`-decoded buffer is sufficient and lighter-weight.
3. If both fail (no theme-color meta, no extractable logo, or `sharp` decode fails on the logo's format) → omit `theme` entirely, falls back to the current neutral template — matches CONTEXT.md's explicit "never breaks" requirement.

**Not evaluated further:** CSS parsing (fetching linked stylesheets, parsing custom properties like `--brand-color`) — technically richer but requires a second/third HTTP fetch per stylesheet, a CSS parser dependency, and highly inconsistent variable-naming conventions across sites (no reliable `--primary`/`--brand` standard). The `theme-color` meta tag + logo-pixel-quantization combination covers the realistic, low-effort cases; CSS parsing is explicitly out of scope for v1 per the "no headless browser beyond what Microlink already provides" framing in the phase brief.

## Failure/fallback behavior (must-have, per CONTEXT.md)

Every new code path in this phase must follow the **exact same shape** as the three existing best-effort scrapers already in this codebase (`lib/email-scrape.ts`, `lib/pagespeed.ts`, `lib/screenshot.ts`): plain `fetch` with an explicit `AbortSignal.timeout(...)`, wrapped in `try { ... } catch { return null/[] }`, never throwing, never surfacing a user-facing error, logged nowhere special (these three don't `console.error` on failure either — a scrape "miss" is treated as a normal, expected outcome, not an error condition).

This means the new logo/photo/color scraping should **not** live inside the `try {} catch { return NextResponse.json({ error }, { status: 502 }) }` block in `app/api/redesigns/generate/route.ts` that surfaces failures to the user (that block is reserved for Places/AI Gateway/Storage failures that *do* block generation). Scraping the original site is **strictly additive and non-blocking**: it should run, swallow its own errors internally (mirroring `captureScreenshot`'s `catch { return null }`), and simply produce `null`/`[]` values that the existing `content` object literal already knows how to handle (since `logoUrl: null` and empty `photoUrls` are already valid states handled by `redesign-preview.tsx` today).

Concretely: `beforeScreenshotUrl` is already computed conditionally on `lead.has_own_website && details.websiteUri` — the new scrape (photos, theme-color) can piggyback on that exact same condition and even the exact same Microlink-provided screenshot data where applicable (logo, at least, comes from the same call). If `has_own_website` is false (no site to scrape), the whole new code path is simply skipped, same as today.

## Scope confirmation (from CONTEXT.md, restated for the plan)

- **New redesigns only** — no backfill/reprocessing job for existing rows. This has zero migration implications since jsonb rows without the new keys are simply read as `undefined`/absent by consuming code (as long as that code treats the new fields as optional, per the "critical finding" section above).
- **Photo rights:** scraping the lead's *own* site for its *own* business's photos is the same product-risk category already accepted for Places Photos (Pitfall 3 in `.planning/research/PITFALLS.md` — "reusing the business's own logo/photos is lower-risk than third-party stock"); CONTEXT.md explicitly reconfirms this is evaluated and accepted by the user, no new legal/rights gate needed for this phase.
- **No AI Gateway call needed** — this is pure extraction/scraping, runs in parallel/alongside the existing `generateRedesignCopy()` call, not a replacement or addition to it.

## Open items for the plan to decide (not blocking, just implementation choices)

- Exact photo-selection heuristic thresholds (pixel-size cutoff, filename denylist, cap count) — reasonable defaults suggested above, tune during implementation/testing against the Dra. Tania Higaki case and 1-2 others.
- Whether to add `sharp` as an explicit dependency now vs. deferring the color-fallback tier entirely to a later pass (theme-color meta tag alone may already cover enough cases to be worth shipping first, with the `sharp` logo-quantization fallback as a fast-follow if theme-color coverage proves too low in practice).
- Whether `theme.primaryColor` should be a single hex string or a small palette (e.g. `{ primary: string; accent?: string }`) — CONTEXT.md says "paleta de cores dominante" (singular dominant color), so a single value is sufficient for v1 and simpler to consume in `redesign-preview.tsx`.
