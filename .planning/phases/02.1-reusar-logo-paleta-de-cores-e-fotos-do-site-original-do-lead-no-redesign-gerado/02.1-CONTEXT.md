# Phase 02.1: Reusar logo, paleta de cores e fotos do site original do lead no redesign gerado - Context

**Gathered:** 2026-07-08
**Status:** Ready for planning
**Type:** INSERTED (urgent gap-closure, inserida após a Phase 2)

<domain>
## Phase Boundary

O redesign gerado (Fase 2) hoje usa só fotos do Google Places + template neutro — não reaproveita nada do visual real do site original do lead (logo, paleta de cores, fotos do hero/equipe). Confirmado num teste manual real em produção (2026-07-08, lead "Dra. Tania Higaki"): o redesign usou fotos de estoque genéricas sem relação com o negócio, enquanto o site original tinha fotos reais da dentista — o usuário achou o resultado "horrível" nesse comparativo lado a lado. Esta fase fecha esse gap: extrair logo, paleta de cores e fotos do site original (quando disponível) e usar no redesign gerado, em vez de só template neutro + Places Photos.

</domain>

<decisions>
## Implementation Decisions

### Elementos a reaproveitar
- Logo, paleta de cores dominante, e fotos reais (hero/equipe) do site original do lead.
- Fotos do Google Places continuam existindo como fonte, mas deixam de ser a única opção — o site original passa a ser preferido quando disponível.

### Fallback
- Deixado a critério técnico (pesquisa/planejamento decide a lógica exata de quando cada elemento não é extraível), mas o resultado observável é: quando não der pra extrair logo/paleta/foto do site original, cai exatamente no comportamento de hoje (template neutro + fotos do Places) — nunca quebra a geração, nunca deixa a página sem visual.

### Escopo (novos vs. existentes)
- Aplica-se só a redesigns gerados **daqui pra frente** — sem migração/reprocessamento de redesigns já existentes. Mesmo padrão já usado na Fase 5 pro campo `facts.phone` (redesigns antigos ficam como estão; o botão "Gerar de novo" já existe pra quem quiser uma versão nova com o scraping).

### Direito de uso das fotos
- Sem restrição adicional além do que já vale hoje: são fotos do PRÓPRIO site do negócio sendo redesenhado pra ele mesmo (não fotos de terceiros) — risco de direito de imagem já avaliado como baixo o suficiente pelo usuário. Fotos do Places continuam como fallback quando o site original não tiver imagens aproveitáveis.

### Claude's Discretion
- Mecanismo técnico exato de extração de logo (ex.: `<link rel="icon">`, meta tags OpenGraph, heurística de imagem no header) e de paleta de cores (ex.: análise de CSS, extração de cor dominante de uma imagem/screenshot).
- Onde no schema `redesigns.content` esses novos dados entram — `types/redesign-content.ts` está marcado como CONGELADO (Anti-Pattern 3, ARCHITECTURE.md); pesquisa deve avaliar se dá pra fazer uma extensão aditiva (novos campos opcionais) sem quebrar o freeze, já que é jsonb e o freeze existe pra proteger contra migração de dados, não contra todo e qualquer campo novo.
- Qual mecanismo de scraping usar (Microlink já é usado pra screenshot "antes" — avaliar se dá pra extrair mais dados dele, ou se precisa de outra abordagem).
- Tratamento de sites que bloqueiam scraping (CORS, anti-bot) — cai no fallback, sem crash.

</decisions>
