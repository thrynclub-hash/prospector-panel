## Risk areas (explicitly re-checked)

1. **`theme` accidentally vulnerable to the photos-wholesale-replace bug** — NOT present. `theme` lives as its own top-level key in `RedesignContent`, separate from `photos`. The PATCH route's `photos: body.photos ?? currentContent.photos` line is untouched and still does whole-object replacement, but `theme` is threaded on its own line (`theme: currentContent.theme`) and is never part of the `photos` object, so it can't be silently dropped by an editor "Save" that reconstructs `photos` by hand.
2. **Scraping code that could throw/block generation** — NOT present. `captureSiteVisuals` and `scrapeSiteAssets` each wrap their entire body in try/catch with typed fallback returns; `uploadFromUrl` also try/catches and returns `null`. All three are called exclusively in the block physically after the error-surfacing `catch`, so even a bug inside one of them causing a thrown error (not currently possible, but hypothetically) would surface as an unhandled 500 on the *scraping* step rather than the existing 502 path — and in practice none of them can throw, confirmed by reading each function body in full.
3. **`redesign-preview.tsx` changes not visually meaningful** — NOT the case. `theme.primaryColor` is consumed in three concrete visual spots (hero background color, "Sobre" heading color, "Serviços" heading color) via inline `style`, which is necessary since the color is dynamic data, not a static Tailwind class. This is real rendering, not a computed-but-unused value.
