# Phase 02.2 Plan 02: Redesign structure elevation + inline editor Summary

**`RedesignPreview` is rewritten to the full `redesign-premium` structure (2-CTA hero, trust bar, WhatsApp-linked service cards, photo gallery, a new Localização e Contato section with Google Maps embed + `facts.openingHours`, floating WhatsApp button) with component-scoped Playfair Display/Inter typography, and gains an optional inline editor (click text to edit, click image to swap) that persists through the existing PATCH/photo-upload routes; `/editar` now renders this real preview in edit mode instead of the old separate form, which is removed.**

## Performance

- **Tasks:** 4 completed
- **Files modified:** 2 modified, 1 created (via pre-existing crash-recovered file), 1 removed

## Accomplishments
- `editable-field.tsx` provides `EditableText` (contentEditable, saves on blur via caller-provided `onSave`) and `EditableImage` (click opens a hidden file input, uploads via `POST /api/redesigns/[id]/photo`, then calls `onUploaded`) -- both degrade to plain, non-interactive rendering when `editable` is false (or `redesignId` is absent, for `EditableImage`)
- `redesign-preview.tsx` fully rewritten as `"use client"`: sticky nav (logo+name+anchor links+WhatsApp CTA), 2-column hero (category label, editable headline/subheadline, 2 CTAs -- WhatsApp solid + "Ver serviços" outline -- editable hero photo with a floating rating card), a trust bar (rating/category/address), a `#servicos` section of service cards that link out to `wa.me` only in read-only mode (in edit mode the card is a plain div so clicks land on the contentEditable text, not a navigation), an about section, a photo gallery (up to 7 additional photos beyond the hero shot, all swappable in edit mode), a `#localizacao` section (Google Maps iframe embed via `?q={address}&output=embed`, no API key; address; phone; `facts.openingHours` list; final WhatsApp CTA) that only renders when there's an address or a phone-derived WhatsApp link, and a `fixed` floating WhatsApp button -- all WhatsApp elements are conditional on `facts.phone` existing, never fabricated
- Typography is scoped to this component only: `Playfair_Display`/`Inter` loaded via `next/font/google`, exposed as `--rp-display`/`--rp-body` CSS variables, applied via inline `style` on every heading (which is the only way to beat the global `h1,h2,h3{font-family:var(--font-display)}` rule in `app/globals.css` without editing that file) -- the panel's own Unbounded/Manrope identity is never used here
- `content.facts` is read via plain destructuring and never wired into `useState`/`EditableText`/`EditableImage` -- verified by grep, no factual field is ever editable; only `content.generated` and `content.photos` are stateful (`useState`) and flow through the editable components when `canEdit` (`editable && Boolean(redesignId)`)
- `editar/page.tsx` rewritten: same Supabase query as before, but renders `<RedesignPreview content={content} editable redesignId={redesign.id} />` instead of `<EditorForm .../>`, with a sticky top bar explaining edit mode -- route/entry point unchanged, so the existing "Editar" button elsewhere keeps working
- `editor-form.tsx` (Fase 3's standalone form) deleted -- confirmed via grep that `editar/page.tsx` (already rewritten) was its only importer
- The 4 read-only call sites (`Comparator`, `/compare`, `/preview`, `/demo/[slug]`) were verified untouched by this plan's commits (`git diff --stat` across all 4 commits shows zero changes to those files) -- they keep passing only `content`, so `editable`/`redesignId` default to `false`/`undefined` and they render the new structure in pure read-only mode
