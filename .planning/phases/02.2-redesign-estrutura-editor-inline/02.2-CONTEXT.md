### O que muda na estrutura da página gerada (RedesignPreview + generate/route.ts)
- Hero: 2 CTAs (WhatsApp sólido cor de marca + "Como funciona" outline), layout 2 colunas em desktop (texto+foto), card de nota flutuante sobre a foto (não barra separada).
- Cards de serviço clicáveis → `wa.me/{phone}?text=Olá! Vim pelo site e quero saber sobre {serviço}` (reaproveitar `toWhatsAppLink` já existente de `lib/proposal/whatsapp-link.ts`, Fase 5).
- Nova seção "Localização e Contato": mapa embed do Google Maps (iframe, sem chave de API extra — embed simples via URL com endereço), endereço, telefone/WhatsApp, horário de funcionamento, CTA final.
- **Horário de funcionamento**: buscar do Places (campo real/verificável do perfil do Maps) — decisão confirmada pelo usuário nesta sessão. Precisa: (a) novo campo no field mask do Places (`regularOpeningHours` ou equivalente — checar nome exato do campo na API Places New e se muda tier de billing, mesmo processo já feito pro telefone na Fase 5/pro logo na Fase 02.1), (b) novo campo em `facts` no schema (`RedesignContent`) — é aditivo, mesma lógica seguraque já validou `theme` como sibling na Fase 02.1.
- Botão WhatsApp flutuante fixo (mobile principalmente) — mesmo padrão do screenshot.
- Nav simples no topo do preview (não precisa de links funcionais complexos, mas visualmente presente).
- Fotos: aumentar o cap de 3 pra mais (regra 2 da skill: "toda foto utilizável") — número exato fica a critério técnico na hora de planejar (ex.: 6-8 como teto razoável, evitar página excessivamente longa).

### Editor inline (substitui/complementa a página `/editar` atual)
- Clicar em texto no preview → vira editável no local (contentEditable), sai do modo edição no blur, salva via PATCH já existente.
- Clicar em imagem → abre seletor de arquivo → upload via rota já existente (`/api/redesigns/[id]/photo`) → atualiza a foto no preview.
- Modelo React, não o script de injeção HTML do plugin original (esse é só a fonte de inspiração de UX, não pra copiar literalmente — Next.js já tem os hooks/estado pra fazer isso de forma mais robusta).
- Página `/editar` atual (Fase 3, formulário simples) — decidir na hora de planejar se é substituída inteiramente pelo modo inline ou se convive como alternativa; a inclinação (não confirmada com o usuário ainda) é substituir, já que o usuário quer "exatamente" a UX do plugin.
