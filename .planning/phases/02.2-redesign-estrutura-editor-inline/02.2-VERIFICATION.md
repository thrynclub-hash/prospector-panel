## Human Verification Required

Everything checkable from static code and a real `npm run build` was verified and is correct. The following require an actual browser/deployed environment and cannot be confirmed by reading code:

1. **Inline text editing end-to-end.** Open `/editar` for a real redesign, click a heading/subheadline/service text/about copy, confirm it becomes editable in place, type a change, click away (blur), and reload the page to confirm the edit actually persisted (i.e. the `PATCH` fired and the DB row updated) — not just that the local state changed.
2. **Inline image swap end-to-end.** Click the hero photo, a gallery photo, and the logo in `/editar`; confirm each opens a native file picker, uploads, shows the `Loader2` spinner during upload, and the new image appears and persists after reload.
3. **Google Maps iframe actually renders.** For a real lead with a real address, confirm the `#localizacao` iframe (`google.com/maps?q=...&output=embed`) loads an actual map centered on the right address, not a blank/error frame (embed-without-API-key can behave inconsistently depending on the exact address string format).
4. **Autosave failure UX.** `persist()` silently swallows network failures by design (state stays "saved" locally even if the PATCH failed) — a human should judge whether this is acceptable UX in practice, or whether a visible "failed to save" indicator is actually needed once real users hit flaky connections.
5. **Visual/responsive check against the reference.** Confirm the hero collapses correctly to 1 column on mobile, the floating rating card doesn't overlap awkwardly at small widths, and the overall look genuinely reads as "the premium redesign the skill describes" — this is inherently a visual judgment call, not a code-correctness one.
6. **Fixed WhatsApp button cosmetic tradeoff.** Per `02.2-CONTEXT.md`'s accepted-discretion decision #3, the `position: fixed` WhatsApp button will visually "stick" to the real browser viewport corner (not the container corner) when `RedesignPreview` is embedded inside `Comparator` (600px box) or `/compare` (50% column). This was accepted as a cosmetic tradeoff rather than fixed — a human should eyeball it once and confirm it's tolerable, since it was never actually rendered and looked at in this session.
7. **Opening hours / photo-cap real data end-to-end** (carried over from `02.2-01`, still unexercised): generate a redesign for a lead whose Google Maps profile declares hours, confirm `facts.openingHours` comes back as a Portuguese string array and renders in `#localizacao`; generate one for a lead with no declared hours and confirm the section simply omits the list (never a fabricated one). Same for confirming a redesign actually pulls up to 8 photos now, not 3. This requires `AI_GATEWAY_API_KEY` configured and a deployed/networked environment — not available in this execution session, same caveat as every other Places/AI-Gateway-dependent phase in this project.
8. **Production deploy + smoke test.** Per `02.2-02-SUMMARY.md`'s own "Next Phase Readiness" note, this phase has not yet been deployed or manually tested in production. A human needs to deploy (`vercel --prod`) and click through `/editar` on a real lead's redesign at least once end-to-end.
